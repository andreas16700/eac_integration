EAC (ΑΗΚ) Home Assistant Integration

Status & engineering notes  ·  v0.1.15  ·  updated 2026-05-30  ·  repo: andreas16700/eac_integration

1. What this integration does

Computes an itemised EAC electricity bill (Tariff 01, residential / low voltage) with net-metering, from two cumulative energy meters, and exposes every bill line item as a sensor. A billing period has a start and end; its bill items are sensors grouped under a device.

Model (the whole thing)

Inputs
Consumption / grid-import meter (= gross imported) + solar export meter (optional), both cumulative kWh.
Offsets
At the period start X, capture two offsets: the consumption reading and the export reading at X (export offset = 0 if no reading at X).
Gross / net at point Y
gross = consumption(Y) − consumption_offset; net = gross − (export(Y) − export_offset) (may be negative on net-export days).
Bill
bill = f(gross, net, period multipliers) — see formulas below.
Multipliers
Per the period's rate month (default = end month): fuel adjustment (¢/kWh, bundled EAC table 2013–2026 + per-month override) and production (€/kWh, tariff default 0.1789, per-month overridable).
ItemBasisFormula
Productionnetnet × production_rate
Fuel adjustmentnetnet × fuel_c/100
RES & EE fundnetnet × 0.005 (outside VAT)
Network usegrossgross × 0.0366 (municipality-specific)
Ancillarygrossgross × 0.0065

VAT—5% × (subtotal incl. fuel+PSO, excl. RES)

Net-metering rule: the offset (gross − net, the export-cancelled portion) is billed for network/ancillary/PSO only — exempt from production, fuel adjustment and RES fund.

3. Approach so far

  • Pure, testable core. billing.py has no HA deps — calculate_bill(gross, net, fuel_c, production_rate, tariff). Unit-tested standalone.
  • Bundled rates. rates_data.json holds the EAC fuel-adjustment table (ΧΑΜΗΛΗ, bimonthly, 2013–2026, revised values), extracted from the official spreadsheet. Per-month overrides for fuel & production live in options.
  • Meter reads. Energy over a window is read from long-term statistics (not raw states), so historical periods work after raw states are purged. Per-bucket value = the meter's recorded state at that bucket minus its reading at the period start.
  • Integration is compute-only (v0.1.15). The integration computes the current bill and exposes the sensors — nothing more. It no longer imports statistics or writes states. Recording the per-period 5-minute history is done entirely by the standalone CLI, so there is exactly one writer and no competing values.

4. History tab vs. statistics — resolved (v0.1.13), then moved to the CLI (v0.1.15)

v0.1.15 update: the state-writing described below was removed from the integration. It now lives only in the standalone CLI (§3 last bullet). Keeping two writers (the integration's live sensor value + the backfill) produced spikes, because the live value used a different calculation than the 5-minute line. One writer (the CLI), aligned to the 5-minute grid, upserting in place, fixes that. The findings below still explain why raw states are required.

Goal: the bill-item sensors must show their full per-period curve — including today, hourly — in the entity History tab.

Resolution (web-researched, then implemented): within the recorder retention window the History tab renders the raw states table and never falls back to statistics — so the only way to show today's curve there is to write real states rows. There is no public API; async_import_statistics writes statistics tables only. v0.1.13 therefore writes today's 5-minute bill values directly into the recorder via the private db_schema models (States/StatesMeta/StateAttributes), on the recorder executor, in an atomic transaction, idempotent by exact-second dedup, reusing the live entity's attributes. A test confirms the rows appear via state_changes_during_period (the History-tab source).

Accepted caveats (documented, deliberate):

  • Uses recorder private APIs — can break across HA schema migrations; pin/track versions.
  • Inserted states are purged after purge_keep_days (~10 days) like any state — so the backfill is re-run every refresh (and old days fall back to the long-term statistics, which we also keep).
  • The reference library ldotlopez/ha-historical-sensor removed this in v3.0 as "fragile"; we accept the fragility because the History-tab display was a hard requirement.

Supported-but-rejected alternatives: a statistics-graph card (durable, but not the History tab and hourly-bucketed); per-entity recorder.purge_entities to force statistics fallback (hourly-averaged, lossy). recorder: exclude does NOT work (it also kills statistics).

Original diagnosis (kept for context):

What's verified in the live recorder right now:

Check (sensor.eac_current_total)Result
Long-term statistics — today (hourly)13 points, 00:00→12:00 ✓ (all 19 sensors)
Long-term statistics — whole period (daily)~80 days back to Mar 12 ✓

STATE rows todaysparse: ~00:00, then a gap, then from ~11:20 (restart)

Root cause (verified, not assumed):

  • The History card/tab draws recorded states for a day when states exist, and falls back to long-term statistics only for days that have no states.
  • Old days have zero state rows → History shows the backfilled statistics → "past daily works".
  • Today has sparse states (and a morning gap, because the integration was down 00:00→11:20 under the broken 0.1.9) → History shows those states, hiding the (correct) backfilled hourly statistics.
So nothing was ever "backfilled as states" — it has always been statistics. Old days display them because they lack states; today does not because it has (sparse) states that take precedence.

Constraint: Home Assistant exposes async_import_statistics for statistics but no public API to inject historical states — which is why option (a), direct states insertion, was implemented in v0.1.12 (above).

5. Issues faced

  • Shipped 0.1.0 without ever loading it in HA. Caused “Config flow could not be loaded: Invalid handler specified” — a stale P_FUEL_OVERRIDE import broke package import so the flow handler never registered.
  • DataUpdateCoordinator now requires config_entry; NumberSelector rejects step<0.001 (use step="any"); DateSelector returns ISO strings (JSON-safe).
  • Repeated model churn / over-engineering. consumption+export → "gross+net direct inputs" → back to consumption+export; a min(gross,export) clamp added then removed; a "monotonic sum" change that wrongly hid legitimate net-export drops.
  • Broken 0.1.9 (renamed config keys → KeyError) took the integration down for hours — that is the source of today's state gap.
  • Behavioral failure: repeatedly declared "done" after checking only that DB rows exist, instead of verifying the actual end-goal (the user seeing the data on the sensors). The remaining History-tab issue went undiagnosed for many iterations because of this.

6. Insights gained (Home Assistant recorder)

  • States cannot be backfilled via any public API; only long-term statistics (async_import_statistics / async_add_external_statistics).
  • History rendering: states-if-present, else long-term statistics. This is why backfilled statistics show for stateless (old/purged) days but not for days with live states.
  • Retention: recorder default purge_keep_days = 10 for states; long-term statistics are hourly and kept indefinitely, aggregated to day/week/month on query.
  • statistics_during_period: period="hour"|"day" reads long-term; "5minute" reads short-term.
  • Energy reads: meter reading = statistics state; energy in a window = state(end) − state(start) (use state, not summed change, which is noisy on real meters).
  • Net metering: net = gross − exported, no clamp; the total legitimately falls (even negative) when exporting more than importing.
  • Data realities on this install: the configured "export" sensor was sensor.solax_total_yield (solar production, not grid export); the source consumption meter's long-term statistics begin only 2026-03-05.
  • Shared calc. The per-point bill (compute_bill_series) and the period-start baseline (reading_at_start, “reading at start else 0”) live in billing.py and are used by BOTH the integration and the standalone CLI (scripts/eac_backfill.py) — identical inputs, identical outputs.
  • Manual CLI. scripts/eac_backfill.py runs inside the HA container, reads the recorder 5-minute statistics, calls the shared compute_bill_series, and idempotently upserts a state at every aligned 5-minute slot (:00,:05,…) for each bill sensor — exactly one value per slot, safe to re-run (connects with CLIENT.FOUND_ROWS so re-runs update in place, never duplicate). Window: --since-hours N (default 1), --this-month (from the 1st of the current month), or --whole-period (from the period start). The offsets are always taken at the period start regardless of window. Backfills with no HA restart.
